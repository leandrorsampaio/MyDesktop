## modal-dialog.js: The Foundation

Open `public/components/modal-dialog/modal-dialog.js`. This is a reusable component that provides the modal "container" — the backdrop, the panel, the close button, the open/close behavior.

**Constructor (lines 5-9):**

```javascript
constructor() {
    super();
    this.attachShadow({ mode: 'open' });
    this._boundHandleEsc = this.handleEsc.bind(this);
}
```

That `bind` line is important. When you add an event listener, you need to be able to remove it later with the exact same function reference. `this.handleEsc.bind(this)` creates a new function every time it's called. So you store the bound version once in the constructor and reuse it for both `addEventListener` and `removeEventListener`.

This is a common source of bugs in vanilla JavaScript. If you write:
```javascript
document.addEventListener('keydown', this.handleEsc.bind(this));
// Later:
document.removeEventListener('keydown', this.handleEsc.bind(this));
// BUG! This is a DIFFERENT function reference. The listener is NOT removed.
```

By storing the bound function, you avoid this trap. In React, this isn't an issue because hooks manage listener lifecycle. In LWC, the same `bind` pattern is used.

**connectedCallback (lines 11-32):**

Standard template loading. After the template is in the Shadow DOM, it sets up three close triggers:

1. Close button click (the X in the corner)
2. Backdrop click (clicking outside the modal panel)
3. ESC key (document-level keydown listener)

For the backdrop click, look at the logic — it checks if the click target is the backdrop element specifically, not any element inside the modal. Clicking on the modal content should NOT close it.

**The `open` Attribute and Lifecycle (lines 38-56):**

```javascript
static get observedAttributes() { return ['open']; }

attributeChangedCallback(name, oldValue, newValue) {
    if (name === 'open') {
        if (newValue !== null) {
            document.addEventListener('keydown', this._boundHandleEsc);
        } else {
            document.removeEventListener('keydown', this._boundHandleEsc);
        }
    }
}
```

When the `open` attribute is added, the ESC listener is attached. When the attribute is removed, the listener is cleaned up. This is reactive behavior driven by attributes — the component responds to its own state changes.

The `open()` and `close()` methods are thin wrappers:

```javascript
open() {
    this.setAttribute('open', '');
}

close() {
    this.removeAttribute('open');
}
```

The CSS in `modal-dialog.css` uses `:host([open])` to show the modal when the attribute is present and hide it when it's absent. So the attribute is doing double duty: it drives both the JavaScript behavior (ESC listener) and the visual state (CSS display).

**disconnectedCallback (lines 48-51):**

```javascript
disconnectedCallback() {
    document.removeEventListener('keydown', this._boundHandleEsc);
}
```

Cleanup. If the modal element is removed from the DOM, make sure the document-level listener is removed too. Without this, you'd have a "ghost listener" — responding to ESC even though the modal is gone. That's a memory leak and a behavior bug.

## The HTML Skeletons in index.html

Now open `index.html` and scroll to the modal sections (lines 50-430).

Look at the task modal (around line 50):

```html
<modal-dialog class="js-taskModal">
    <span slot="title">Add Task</span>
    <form class="taskForm js-taskForm">
        <input class="js-taskTitle" type="text" maxlength="200" required>
        <!-- ... more fields ... -->
    </form>
</modal-dialog>
```

Notice `<span slot="title">Add Task</span>`. This is the **slot** mechanism. The modal-dialog template has a `<slot name="title">` inside it. The content you put in `<span slot="title">` gets projected into that slot. The modal component provides the frame (backdrop, close button, panel); the content is injected from outside.

This is identical to LWC slots. In LWC, you'd write `<c-modal-dialog><span slot="title">Add Task</span></c-modal-dialog>` — exactly the same pattern.

Also notice: the form fields are all here in the HTML, not generated by JavaScript. The `js-` hooks (`.js-taskTitle`, `.js-taskDescription`, `.js-taskPriority`) allow JavaScript to read and write their values. But the structure is static.

Scroll through the other modals. They all follow the same pattern: `<modal-dialog>` wrapper, a `js-` class for identification, content inside with `js-` hooks.

The small confirmation modals (around lines 149-284) are interesting — they have a message element and two buttons (Cancel + Action):

```html
<modal-dialog class="js-confirmModal" size="small">
    <span slot="title">Confirm Delete</span>
    <p class="js-confirmMessage"></p>
    <div class="modalActions">
        <custom-button class="js-confirmCancel" modifier="secondary">Cancel</custom-button>
        <custom-button class="js-confirmAction" modifier="danger">Delete</custom-button>
    </div>
</modal-dialog>
```

The message is set dynamically by JavaScript. The buttons have listeners attached by the modal orchestration code.

## modals.js: The Orchestration Layer

Now open `public/js/modals.js`. At nearly 2000 lines, this is the largest module. But it's organized into clear sections.

**The Task Modal Logic (lines 72-400)**

Look at `openAddTaskModal()` around line 184:

```javascript
export function openAddTaskModal(elements, onDelete, onSubmit) {
    elements.taskModal.querySelector('[slot="title"]').textContent = 'Add Task';
    elements.taskForm.reset();
    // ... clear previous values
    // ... set up category pills
    // ... set up epic picker
    renderTaskModalActions(false, elements, onDelete, onSubmit);
    elements.taskModal.open();
}
```

It sets the title, resets the form, configures the UI elements, and calls `.open()`. The modal component handles the visual opening. This function handles the content.

Now look at `openEditModal()` around line 213. Almost the same, but it loads existing task data into the form:

```javascript
export function openEditModal(taskId, elements, onDelete, onSubmit, onClone, onSendToBacklog) {
    const task = findTask(taskId);
    elements.taskModal.querySelector('[slot="title"]').textContent = 'Edit Task';
    elements.taskTitle.value = task.title;
    elements.taskDescription.value = task.description || '';
    elements.taskPriority.checked = task.priority;
    // ... set category, epic, deadline, snooze
    // ... render activity log
    renderTaskModalActions(true, elements, onDelete, onSubmit, onClone, onSendToBacklog);
    elements.taskModal.open();
}
```

Same modal element, different content. This is modal reuse — one DOM element, multiple modes (Add / Edit / Clone).

**`renderTaskModalActions()` (lines 84-130):**

This function dynamically builds the button bar at the bottom of the task modal. In Add mode, it shows Cancel and Save. In Edit mode, it shows Cancel, Clone, Backlog, Update, and a delete button. The Backlog button only appears for non-backlog tasks.

Each button is a `<custom-button>` element with a `modifier` attribute for styling. Event listeners are attached with the `js-` hooks.

**The form submission handler is `createTaskFormSubmitHandler()` (lines 323-400):**

This is a **factory function** — a function that creates and returns another function. It's called once during initialization and returns the handler that's used for every form submit.

```javascript
export function createTaskFormSubmitHandler(elements, renderColumn, renderAllColumns, addTaskToState, updateTaskInState, overrideColumnId) {
    return async function handleSubmit(e) {
        e.preventDefault();
        // ... collect form data
        // ... optimistic update
        // ... API call
        // ... rollback on error
    };
}
```

Why a factory? Because the handler needs access to several functions (`renderColumn`, `addTaskToState`, etc.) that come from different modules. The factory captures them in a closure. The returned handler can use them without needing global access.

This is the **closure** pattern — one of JavaScript's most powerful features. The inner function "closes over" the variables from the outer function. Even after `createTaskFormSubmitHandler` finishes executing, the returned handler still has access to `elements`, `renderColumn`, etc.

The handler itself follows the optimistic pattern:
1. Read form values
2. Create task with temp ID (or update existing task)
3. Add to state and render immediately
4. Close the modal
5. API call in background
6. On success: replace temp task with server response
7. On failure: remove temp task (or rollback update), toast error

## The Epics Modal: A Complex Example

Look at `openEpicsModal()` around line 595. This is more complex because it manages a full CRUD interface inside the modal.

Instead of having a separate component for the epics editor, the code builds the HTML directly using template literals and `innerHTML`. Then it queries for `js-` hooks and attaches listeners.

```javascript
export function openEpicsModal(elements, closeMenu, onEpicsChanged) {
    closeMenu();
    const modal = elements.epicsModal;
    const container = modal.querySelector('.js-epicsContent');

    container.innerHTML = `
        <div class="epicsEditor">
            <div class="epicsEditor__list js-epicsList"></div>
            <div class="epicsEditor__form">
                <!-- form fields -->
            </div>
        </div>
    `;

    // Wire up event listeners
    const list = container.querySelector('.js-epicsList');
    const form = container.querySelector('.js-epicsForm');
    // ... add, edit, delete handlers
}
```

This is the "render into container" pattern — an alternative to creating a Web Component. For UI that only appears inside a modal and isn't reused elsewhere, building a component would be over-engineering. The modal IS the container; the JavaScript renders and manages the content directly.

The callback `onEpicsChanged` is called whenever epics are modified. This allows the calling code (`app.js`) to refresh the board's epic data without the modal needing to know about the board.

## The Confirmation Pattern

Search for `openDeleteConfirmation()` around line 406:

```javascript
export function openDeleteConfirmation(elements) {
    pendingDeleteId = editingTaskId;
    elements.confirmMessage.textContent = 'Are you sure?';
    elements.confirmModal.open();
}
```

And the action handler:

```javascript
export function confirmDeleteTask(elements, renderAllColumns, removeTaskFromState) {
    if (!pendingDeleteId) return;
    // ... optimistic delete
    elements.confirmModal.close();
    pendingDeleteId = null;
}
```

The pattern: store the pending operation's data in a module-level variable (`pendingDeleteId`), open the confirmation modal, wait for the user to click the action button, execute the operation, clear the pending state.

Why not use `confirm()`? Because it blocks the browser thread. The page freezes until the user clicks OK or Cancel. It's ugly (native OS dialog, can't be styled) and shows the site URL (which alerts users that it's a JavaScript dialog, breaking immersion). The custom modal looks better, behaves better, and follows the same component pattern as everything else.

## Key Takeaway

Your modal system has three layers: the `<modal-dialog>` component (open/close/ESC/backdrop), the HTML skeletons in `index.html` (structure and hooks), and `modals.js` (orchestration, data flow, CRUD logic). The component doesn't know what content it holds. The HTML provides the structure. The JavaScript makes it interactive.

This separation would look different in React (modal state in useState, content as JSX, portal for rendering) or in LWC (modal as a component with `@api` methods, content via slots). But the concept — a reusable container with injected content — is universal.
